# 自動化通知系統

## 自動化通知與 Webhook 觀念
在開發任何自動化工具（如股市盯盤、客戶表單收集、伺服器狀態監控）時，我們首先要面對的問題是：系統如何把最新的資料傳遞給使用者或另一個系統？

常見的模式可以分為兩種：「主動查看（Pull）」與「被動接收（Push）」。
1. 主動查看 (Pull / 輪詢 Polling)
   想像一下，你正在等待一封非常重要的實體錄取通知信。為了確認信到了沒，你每隔 5 分鐘就走到樓下的郵箱查看。如果信還沒來，你就白跑一趟；如果信來了，你最多可能也延遲了 5 分鐘才看到。
   在數位世界中，這叫做 輪詢 (Polling)。
   - 情境：你寫了一個程式，每隔 1 分鐘就發送 API 請求去檢查 Google Sheets 有沒有新的一列資料，或者某隻股票價格有沒有突破 100 元。
   - 缺點：
       - 浪費資源：99% 的請求可能都是「資料無變更」，這白白消耗了你的網路頻寬、CPU 算力，以及 Google API 的調用配額（Quota）。
       - 延遲（Latency）：如果變更發生在第 2 鐘，而你的檢查間隔是 5 分鐘，這意味著你最久需要等待 3 分鐘才能得知最新消息，無法做到真正的「即時」。
2. 被動接收 (Push / 事件驅動 Event-Driven)
   現在換個場景：你不再每 5 分鐘下樓看郵箱，而是裝了一個智慧郵箱。當郵差把信投入郵箱時，郵箱上的感應器立刻觸發，並發送一則 App 訊息到你的手機。你只需要坐在沙發上，收到通知再下樓即可。
   這就是 事件驅動 (Event-Driven) 的即時警報系統。
   - 情境：你不需要主動去問 Google Sheets 有沒有新資料。相反地，當有人填寫表單並寫入 Google Sheets 的那一瞬間（事件發生），Google 的伺服器會主動發送一則訊息「告訴」你的系統。
   - 優點：
       - 極致省資源：沒有事件發生時，雙方伺服器都處於靜止狀態，資源消耗為零。
       - 高即時性：事件發生與接收通知之間的延遲通常在毫秒到秒級之間。

###  SMTP 協定
在 Webhook 之前，我們必須先理解網際網路上最古老、最穩健的主動通知技術：SMTP (Simple Mail Transfer Protocol，簡單郵件傳輸協定)。

#### 1. 什麼是 SMTP？
SMTP 是網際網路應用層的標準協定之一，專門用於發送（或轉發）電子郵件。需要特別注意的是，SMTP 只管「送信」，當郵件到達目的地伺服器後，收件人要閱讀郵件則需要透過其他協定（如 POP3 或 IMAP）。

#### 2. 傳統郵件傳輸的運作方式
一封電子郵件從你的程式發出到客戶看見，通常經歷以下步驟：

1. MUA (Mail User Agent)：這是郵件客戶端（例如你的 Python 寄信腳本、Outlook 或 Gmail 網頁介面）。你在這裡撰寫信件並按下發送。
2. MSA / MTA (Mail Transfer Agent)：你的腳本會透過 SMTP 協定，將信件連線並推送到你指定的 SMTP 伺服器（例如 smtp.gmail.com）。這個過程需要身份驗證（帳號密碼或 App 密鑰）。
3. DNS 查詢：發信伺服器看到收件人是 user@example.com，它會去查詢 example.com 的 MX 紀錄 (Mail Exchanger)，找出負責接收該網域郵件的目標伺服器 IP。
4. MTA 之間傳遞：發信伺服器透過 SMTP 協定，將信件傳送給目標 SMTP 伺服器。
5. MDA (Mail Delivery Agent)：目標伺服器收到信後，將信件分類放入收件人的電子墓（Mailbox）中。

#### 3. SMTP 的指令與互動範例
SMTP 是一個純文字的交談式協定。當你的程式與 SMTP 伺服器建立 TCP 連線（通常是 Port 25, 465 或 587）後，背後的對話其實長這樣：
```
S: 220 smtp.example.com ESMTP Postfix
C: HELO myclient.com
S: 250 smtp.example.com, I am glad to meet you
C: MAIL FROM:<sender@example.com>
S: 250 2.1.0 Sender ok
C: RCPT TO:<receiver@example.com>
S: 250 2.1.5 Recipient ok
C: DATA
S: 354 End data with <CR><LF>.<CR><LF>
C: Subject: 自動化系統測試通知
C: From: sender@example.com
C: To: receiver@example.com
C: 
C: 您好，這是來自自動化系統的即時警報。
C: .
S: 250 2.0.0 Ok: queued as 12345
C: QUIT
S: 221 2.0.0 Bye
```
(註：C: 代表 Client 客戶端程式，S: 代表 Server 郵件伺服器)

#### 4. 現代自動化通知中的 SMTP 角色
雖然電子郵件看起來很傳統，但在自動化通知中，它依然不可或缺：
- 優點：格式通用（支援 HTML/附件）、非同步閱讀（不打擾使用者）、適合留存稽核紀錄（Audit Log）。
- 缺點：設定繁瑣（需要處理 SPF, DKIM, DMARC 等防垃圾信機制）、即時性相對較低（使用者不會每秒盯著信箱）。

### Webhook 原理
隨著 Web 應用與 API 的普及，我們需要一種更輕量、更符合現代網頁架構（HTTP/JSON）的即時通知機制，這就是 Webhook。

#### 1. 什麼是 Webhook？
如果說 API 是「客戶端主動呼叫伺服器以獲取資料（Pull）」，那麼 Webhook 就是「反向 API」。

Webhook 是一種客製化 HTTP 回呼（HTTP Callback）。當特定事件在源頭伺服器（Provider）發生時，該伺服器會主動向你預先指定的網址（URL）發送一個 HTTP POST 請求，並將事件的詳細資料以 JSON 格式封裝在 Request Body 中。

#### 2. Webhook 運作機制
要讓 Webhook 運作，必須包含以下三個要素：

1. 事件源 (Event Provider)：擁有資料並會觸發事件的平台（例如：TradingView 股價突破、GitHub 有人 Push 程式碼、Google Forms 有新回覆）。
2. 接收端 URL (Webhook URL / Endpoint)：由你（開發者）提供的一個公開網頁網址，這個網址背後運作著一個 Web 伺服器，專門用來監聽並解析傳入的 HTTP 請求。
3. Payload (資料載體)：事件發生時傳送的具體資料，現今標準幾乎全數採用 JSON 格式。

#### 3. Webhook 的即時 JSON 資料範例
當一個「股價突破」的事件被觸發時，Webhook 伺服器會向你的 URL 發送一個類似以下的 HTTP POST 請求：
```
POST /api/my-webhook-receiver HTTP/1.1
Host: yourdomain.com
Content-Type: application/json
User-Agent: Webhook-Provider-Bot
X-Webhook-Signature: sha256=9f86d081884c7d659a2feaa0c55ad015a3bf4f1b2b0b822cd15d6c15b0f00a08

{
  "event": "price_alert",
  "timestamp": 1779931706,
  "data": {
    "ticker": "AAPL",
    "company_name": "Apple Inc.",
    "current_price": 182.50,
    "condition": "cross_above",
    "target_price": 180.00,
    "triggered_by": "Strategy_Alpha"
  }
}
```
欄位解析：
- Content-Type: application/json：告知接收端，這是一段 JSON 格式的資料。
- X-Webhook-Signature：安全簽章（極度重要）。用來驗證這封請求確實是由官方發出，而不是駭客偽造的惡意請求。
- data 物件：包含了這次事件的所有細節（哪個代號、當前價格、觸發條件），你的程式可以直接讀取這些欄位並做後續處置（例如下一單買進或傳送 Line 訊息）。

### 實際應用場景
場景 A：電商網站系統管理員希望在「伺服器硬碟空間不足 10%」時獲得通知
- 最佳選擇：SMTP (電子郵件) 或 Webhook 串接 Slack/Line。
- 考量：這類通知不需要微秒級的反應，但需要嚴肅對待。發送一封郵件可以同時抄送給整個 IT 團隊，且方便在信件中附帶詳細的伺服器日誌（Log）報告。

場景 B：量化交易程式需要在「股價突破壓力線」時立刻下單
- 最佳選擇：Webhook。
- 考量：金融交易分秒必爭。透過 Webhook，交易訊號可以在 50 毫秒內從分析平台投遞到你的自動化交易機器人，機器人讀取 JSON 後能立馬調用券商 API 下單。如果改用輪詢或看 Email，機會早就溜走了。

場景 C：問卷系統在「客戶填寫完滿意度調查」後自動將資料寫入公司內部的 CRM 系統
- 最佳選擇：Webhook。
- 考量：客戶什麼時候填問卷是完全不可預測的。如果用輪詢，系統可能一天要做 1440 次做白工的檢查；使用 Webhook，只有在客戶按下「提交」的那一刻才會觸發一次精準的資料傳輸。

## Gmail 自動化：GAS 與 Python 寄信實務與資安
首先使用 Google 生態系最直覺的 Google Apps Script (GAS) 結合 Google Sheets，實現台股價格觸及監控與自動寄信；接著進階到 Python (Google Colab) 環境，利用 yfinance 抓取即時數據，並重點教學如何利用 Colab 的 Secrets 鍵值管理功能，保護你的應用程式密碼，杜絕資安外洩風險。

### GAS 整合 Google Sheets 發送台股監控通知
Google Apps Script (GAS) 是基於 JavaScript 的雲端腳本語言，它原生整合了 Google 所有服務。當我們想要針對試算表（Google Sheets）內的資料進行邏輯判斷（例如：股價超過設定的目標價）並觸發動作時，使用 GAS 的 MailApp 或 GmailApp 服務是最快、最穩定的解決方案。

這種架構的優點在於無需部署額外的伺服器，只要在 Google 雲端硬碟上就能完成所有工作，且完全免費（在每日配額內）。

#### 範例實作：台股價格觸及監控系統
請想像你正在監控一張台股清單，當某檔股票的「當前價格」大於「設定目標價」時，系統必須立刻寄信通知。

[GAS 股價監控與自動寄信](https://docs.google.com/spreadsheets/d/1lPbRljG93oiIGFujpvZk238hRl-Pql003LukzFjaP-Q/edit?usp=sharing)

確認欄位順序如下：
- A欄：股票代號 (e.g., 2330)
- B欄：股票名稱 (e.g., 台積電)
- C欄：當前股價 (可以使用 =GOOGLEFINANCE("TPE:"&A2) 函數自動連動)
- D欄：目標警告價 (e.g., 1000)
- E欄：通知狀態 (若已寄信，程式會在此寫入 "已發送"，避免重複寄信)
點選選單的 「擴充功能」 -> 「Apps Script」，清空原本的程式碼，並貼上以下指令碼：

```
function checkStockAndSendMail() {
  // 1. 取得當前試算表與工作表
  var sheet = SpreadsheetApp.getActiveSpreadsheet().getSheetByName("台股監控");
  if (!sheet) {
    Logger.log("找不到名為 '台股監控' 的工作表！");
    return;
  }
  
  // 2. 取得資料範圍 (從第 2 行開始，讀取到有資料的最後一行)
  var lastRow = sheet.getLastRow();
  if (lastRow < 2) {
    Logger.log("沒有足夠的資料進行監控。");
    return;
  }
  
  var range = sheet.getRange(2, 1, lastRow - 1, 5); // 讀取 A2 到 E(lastRow)
  var data = range.getValues();
  
  // 3. 設定接收通知的電子郵件（請替換為你的 Gmail）
  var emailRecipient = "your_email@gmail.com"; 
  
  // 4. 逐行檢查股價
  for (var i = 0; i < data.length; i++) {
    var stockId = data[i][0];    // A欄：股票代號
    var stockName = data[i][1];  // B欄：股票名稱
    var currentPrice = data[i][2];// C欄：當前股價
    var targetPrice = data[i][3]; // D欄：目標警告價
    var status = data[i][4];      // E欄：通知狀態
    
    // 判斷邏輯：當前價格 >= 目標價，且尚未發送過通知
    if (currentPrice >= targetPrice && status !== "已發送") {
      
      // 定義郵件主旨與內容
      var subject = "【台股警報】" + stockName + " (" + stockId + ") 已觸及目標價！";
      var body = "您好：\n\n您監控的股票 " + stockName + " (" + stockId + ") " +
                 "目前價格已達 " + currentPrice + " 元，超過您設定的目標價 " + targetPrice + " 元。\n" +
                 "請即時確認市場行情。\n\n本信件由 ytlAi Lab 自動化系統發送。";
      
      try {
        // 發送 Gmail 郵件
        GmailApp.sendEmail(emailRecipient, subject, body);
        
        // 更新試算表狀態，避免下次執行時重複寄信 (i+2 是因為索引從 0 開始且跳過標題列)
        sheet.getRange(i + 2, 5).setValue("已發送");
        Logger.log("成功發送通知：" + stockName);
        
      } catch (error) {
        Logger.log("發送失敗，錯誤訊息：" + error.toString());
      }
    }
  }
}
```
部署與執行說明：
1. 修改 emailRecipient 為你自己的 Gmail 地址。
2. 點擊上方的「儲存」圖示，並點擊「執行」。
3. 初次執行時，Google 會要求授權。 請點選「進階」->「前往『未命名專案』（不安全）」並允許存取。這是因為腳本需要權限來讀取你的試算表並代表你寄出郵件。
4. 若想做到全自動化，可點擊左側時鐘圖示（觸發條件），新增一個「時間驅動」的觸發程序，設定為「每小時」或「每分鐘」執行一次。

### Python yfinance + Gmail 寄信與環境變數資安防範
當自動化需求變得更複雜（例如需要結合機器學習模型預測、複雜的資料清洗或跨平台整合）時，我們通常會切換到 Python 環境。使用 Google Colab，透過 yfinance 庫抓取台灣股市數據，並利用內建的 smtplib 模組透過 Gmail 的 SMTP 伺服器寄信。

#### 資安問題：硬編碼（Hardcoding）
在開發過程中，最常看見初學者犯下致命的資安錯誤：直接將 Gmail 帳號與密碼以字串形式寫在程式碼中。
```
#  錯誤示範！千萬不要這樣寫！
password = "my_secret_password_1234"
```
為什麼這很危險？

1. GitHub 慘劇：當你高興地將專案打包，並使用 git push 上傳到 GitHub 公開儲存庫時，全世界的機器人（Bot）會在幾秒鐘內掃描到你的密碼。你的 Gmail 將被用來發送大量垃圾信，甚至導致 Google 帳號遭到永久封鎖。
2. 共用風險：當你將 Colab 筆記本分享給同學或同事協助除錯時，密碼毫無保留地暴露在他人眼前。

#### Google 帳號產生「應用程式密碼」
為什麼不能直接用 Gmail 的登入密碼？因為 Google 為了保護帳號安全，預設會阻擋任何第三方腳本（如 Python）直接使用你的主密碼登入。因此，必須產生一組專門給 Python 程式使用的「應用程式密碼」（App Password）。這是一組獨立的 16 位元字母密碼，隨時可以單獨刪除，不會影響你的主密碼。
1. 前往 Google 帳號管理中心：
   - 瀏覽器開啟並登入 Google 帳號管理中心。
2. 開啟雙重驗證（必要條件）：
   - 點選左側選單的 「安全性」。
   - 找到「登入 Google 的方式」區塊，確認 「兩步驟驗證」 是否已開啟。
   - 註：如果沒開啟，請依畫面提示綁定手機或驗證碼App來啟用它。未開啟兩步驟驗證將無法產生應用程式密碼。
3. 尋找應用程式密碼：
   - 開啟兩步驟驗證後，直接點進去 「兩步驟驗證」 的設定畫面。
   - 頁面拉到最下方，你會看到 「應用程式密碼」（App passwords）這個選項，點擊進入。
4. 產生密碼：
   - 在「應用程式名稱」輸入一個好辨識的名字，例如：Colab Stock Bot 或 ytlAi Lab Test。
   - 點擊 「建立」。
5. 複製密碼：
   - 畫面上會跳出一個黃色/設定框，裡面有一組 16 位元的英文隨機密碼（例如：abcd efgh ijkl mnop）。
   - 請立刻把它複製下來（可以先貼在記事本中，中間的空格不用刪除），因為這個密碼只會顯示這一次，關閉視窗後就再也看不到了！

#### 使用 Google Colab Secrets 功能
現代雲端開發環境提供了金鑰管理系統（Secret Management）。在 Google Colab 中，左側工具列有一個 「鑰匙（Secrets）」 圖示。我們應該將敏感資料（如 API Key、密碼）儲存在這裡，它會以加密的形式保存在你的 Google 帳號中，而不會出現在程式碼內。

設定步驟：
1. 點擊 Colab 左側的 🔑 Secrets（鑰匙圖示）。
2. 點擊 "Add new secret"。
3. Name (鍵) 輸入：GMAIL_USER，Value (值) 輸入你的 Gmail 帳號。
4. 再新增一個，Name (鍵) 輸入：GMAIL_PASSWORD，Value (值) 輸入你的 Google 應用程式密碼（注意：非 Gmail 登入密碼，需至 Google 帳號安全性中開啟雙重驗證並產生「應用程式密碼」）。
5. 務必將這兩個 Secret 的 "Notebook access" 勾選開啟，允許當前的筆記本讀取它。

import smtplib
from email.mime.text import MIMEText
from email.header import Header
import yfinance as yf
from google.colab import userdata

def send_secure_email(stock_code, current_price, target_price):
    try:
        gmail_user = userdata.get('GMAIL_USER')
        gmail_password = userdata.get('GMAIL_PASSWORD')
    except Exception:
        print("❌ 郵件發送失敗：讀取 Secrets 失敗，請確認左側鑰匙設定！")
        return

    smtp_server = "smtp.gmail.com"
    smtp_port = 465

    mail_content = f"""
    您好：
    
    【yfinance 批次監控警報】
    監控標的 {stock_code} 目前股價已達 {current_price} 元，
    已成功突破您設定的警戒線 {target_price} 元。
    
    系統已自動記錄此狀態。
    """
    
    message = MIMEText(mail_content, 'plain', 'utf-8')
    message['From'] = Header(f"yfinance 監控中心 <{gmail_user}>", 'utf-8')
    message['To'] = Header(gmail_user, 'utf-8')
    message['Subject'] = Header(f"🚨 批次觸及警報：{stock_code}", 'utf-8')

    try:
        with smtplib.SMTP_SSL(smtp_server, smtp_port) as server:
            server.login(gmail_user, gmail_password)
            server.sendmail(gmail_user, [gmail_user], message.as_string())
        print(f"📧 [信件發送] 標的 {stock_code} 通知信已成功送達。")
    except Exception as e:
        print(f"❌ 傳送失敗: {e}")

# --- 綜合挑戰主程式 ---
def batch_monitor():
    # 定義監控清單字典 { "股票代碼": 目標警戒價 }
    # 💡 為了方便測試，你可以故意把目標價設得比市價低，確保能觸發寄信邏輯
    monitor_list = {
        "2330.TW": 950.0,   # 台積電
        "2317.TW": 150.0,   # 鴻海
        "2454.TW": 1300.0   # 聯發科
    }
    
    print("🚀 啟動 yfinance 多檔標的批次監控系統...")
    print("-" * 50)
    
    for stock_id, target_price in monitor_list.items():
        print(f"🔍 正在檢查 {stock_id}...")
        try:
            # 抓取即時數據
            stock = yf.Ticker(stock_id)
            todays_data = stock.history(period="1d")
            
            if todays_data.empty:
                print(f"❌ 無法取得 {stock_id} 的數據，跳過此標的。")
                continue
                
            # 取得當前最新收盤價
            latest_price = round(todays_data['Close'].iloc[-1], 2)
            print(f"📊 當前市價: {latest_price} 元 | 設定目標: {target_price} 元")
            
            # 條件判斷
            if latest_price >= target_price:
                print(f"⚠️ 觸發警報！市價 {latest_price} >= 目標價 {target_price}")
                send_secure_email(stock_id, latest_price, target_price)
            else:
                print(f"✅ 安全。未達警告門檻。")
                
        except Exception as error:
            print(f"💥 處理 {stock_id} 時發生非預期錯誤: {error}")
            
        print("-" * 50)
        
    print("🏁 所有標的檢查完畢。")

# 執行批次監控
batch_monitor()
